## 2. Duplicate title, 150 grup (prod)

150 farklı başlık birden fazla `slug` ile kayıtta. Dağılım:
- **140 grup × 2 kopya** = 280 tarif (her grup 1 fazla row)
- **10 grup × 3 kopya** = 30 tarif (her grup 2 fazla row)
- **Toplam fazlalık: 160 row**

Örnekler (prod):

```
kedgeree × 3     → [kedgeree, kedgeree-ingiliz-usulu, kedgeree-ingiliz-kahvalti-usulu]
kladdkaka × 3    → [kladdkaka, kladdkaka-isvec-usulu, kladdkaka-portakalli-isvec-usulu]
mizeria × 3      → [mizeria-polonya-ev-usulu, mizeria-polonya-usulu, mizeria-polonya-yaz-usulu]
nyponsoppa × 3   → [nyponsoppa, nyponsoppa-bademli-isvec-usulu, nyponsoppa-isvec-usulu]
picarones × 3    → [picarones, picarones-mini-peru-usulu, picarones-peru-usulu]
sellou × 3       → [sellou, sellou-fas-usulu, sellou-kuzey-afrika-usulu]
```

Her kopya hemen hemen aynı `cuisine` + benzer ingredients + benzer
description (sadece prep süresi, ingredient sayısı ve detay tonu
değişiyor). SEO ve UX etkisi:

- Kullanıcı arama yaptığında 3 "Nyponsoppa" çıkıyor, hangisi "gerçek"
  anlaşılmıyor (seçim paradoksu)
- Google "thin content / duplicate content" sinyali → ranking kaybı
- Sitemap'te aynı içerik 3 URL ile yayınlanıyor (crawl budget israfı)
- JSON-LD Recipe.aggregateRating 3 ayrı entity'ye bölünüyor

Fix stratejisi (3 adım, ayrı iş paketleri):

### 2.1 Winner/loser analizi (TAMAMLANDI, prod'dan)

Script: `scripts/audit-duplicate-winner.ts`, read-only. CSV çıktısı
`docs/duplicate-title-plan-2026-04-20.csv` (150 grup, 160 loser row).

Winner heuristic (score formülü):
- `isFeatured` dominant (+1000)
- `ingredientCount × 10` + `stepCount × 5`
- `description.length × 0.1`
- `viewCount × 0.01`
- Kısa slug bonusu (`slug.length × -0.1`)

**Kritik bulgu:** **0 loser reference to migrate** (Bookmark + CollectionItem
+ MealPlanItem + Variation + Review, hepsi 0). Prod'da hiçbir loser tarife
henüz user-generated content bağlanmamış. **Merge tamamen güvenli, veri
kaybı yok.**

Örnek kararlar (ilk 10):

| Grup | Winner | Losers | Cuisine |
|---|---|---|---|
| kedgeree × 3 | `kedgeree-ingiliz-kahvalti-usulu` (featured) | kedgeree, kedgeree-ingiliz-usulu | same (gb) |
| kladdkaka × 3 | `kladdkaka-portakalli-isvec-usulu` (featured) | kladdkaka, kladdkaka-isvec-usulu | same (se) |
| picarones × 3 | `picarones` (6 ing, kök) | picarones-peru-usulu, picarones-mini-peru-usulu | same (pe) |
| toyga-corbasi × 3 | `toyga-corbasi` (8 ing, kök) | toyga-corbasi-ankara-usulu, toyga-corbasi-corum-usulu | same (tr) |

**Kerem onayı gerekli:** Winner heuristic isFeatured'ü dominant alıyor
ama bazı grupta kök slug (`picarones`, `toyga-corbasi`) SEO için daha
iyi olabilir. Alternatif: "isFeatured yoksa kök slug tercih" politikası
eklenebilir. Şu an CSV rapor için hazır, Kerem gözden geçirir.

### 2.2 Title rename alternatifi (destructive olmayan)
Bazı duplicate'ler gerçek bölgesel varyant olabilir:
- "Kladdkaka" → "İsveç Kladdkakası" (kök), "İsveç Usulü Kladdkaka" (geleneksel),
  "Portakallı Kladdkaka" (mod): title'lar benzersiz, slug'lar korunur
- Ama `mizeria-polonya-ev-usulu` / `mizeria-polonya-usulu` / `mizeria-polonya-yaz-usulu`
  üçünde de `cuisine=pl`, Polonya salatası içeriği benzer → farklılaşma yok,
  silme tercih edilir

### 2.3 Destructive merge (Kerem onayı sonrası)
- Script: `scripts/merge-duplicate-recipes.ts` (yeni)
- Her grup winner + loser listesi CSV'den okunur
- Transaction:
  - Loser'ların `Bookmark` + `CollectionItem` + `MealPlanItem` referansları
    winner'a migrate (ON CONFLICT DO NOTHING, composite unique)
  - Loser `Recipe` delete cascade (RecipeIngredient, RecipeStep, RecipeTag,
    Variation, Review, ViewDaily cascade tetikler)
- `seed-recipes.ts` source patch (loser block'ları kaldır) → idempotent
- Prod'da `--confirm-prod` ile çalıştır

**Tahmini kazanç:** 160 fazla row silinir → storage ~500 KB, sitemap
160 URL azalır, Google duplicate content penalty riski kaybolur,
unique title sayısı 150 (300 distinct konsept değil 150 + 20 gerçek
varyant).

## 3. Dev-prod-source drift, 3 katmanlı sapma

Gerçek durum ilk varsayıldığından karışık:

| Katman | Tarif sayısı | Fark (prod baz) |
|---|---|---|
| `scripts/seed-recipes.ts` | **2286** | prod'dan **34 eksik** |
| Dev DB | **2301** | prod'dan 19 eksik |
| Prod DB | **2320** | baz |

Yani source of truth divergent: seed source hem dev hem prod'un gerisinde;
dev prod'un gerisinde. Seed'i olduğu gibi koşmak drift'i kapatmıyor
(denendi, 0 yeni tarif eklendi, 2286 hepsi "zaten var"), çünkü eksik
tarifler kaynak dosyada **yok**.

**Sebep:** Bazı batch allergen fix'leri ve drift-fix satırları prod'a
direkt DB patch ile uygulanmış (seed regenerate akışı atlanmış), kaynak
dosya bu 34 tarifi görmeden kaldı. Dev'e de kısmi kopya (19 tarif) geçmiş
ama 15 tanesi dev'de de yok.

**Önerilen 2-adım fix (ayrı iş paketi):**

1. Prod'dan DB → seed-recipes.ts sync: `scripts/patch-source-from-db.ts`
   mevcut, ancak bu SADECE var olan tarifi günceller. Eksik tarifleri
   **ekleyen** varyantı gerekli (yeni script veya patch-source
   `--include-missing` mode).
2. Dev'i prod'dan senkron et: `scripts/seed-recipes.ts` çalıştır (source
   2320'ye güncel olduktan sonra idempotent upsert eksik 19'u ekler).

**Şimdilik:** 34 tarif prod'da canlı, SEO + kullanıcı deneyimi etkilenmiyor.
Kritik değil ama source truth'u hizaya çekmek uzun vadede mantıklı. Kerem
onayı ile ayrı tur.

### 5.3 Unused index (Neon stats reset sonrası)

15+ index `idx_scan=0` raporlanıyor. Bu Neon'un son `pg_stat_reset()`
sonrası henüz sorgulanmamışlar demek, kalıcı "unused" değil. Launch plan'a
geçerken veya son `prisma migrate dev` sırasında reset olmuş olması
muhtemel. **Silmeden önce bir hafta bekleyip tekrar audit etmek gerek**
(prod üzerinde gerçek trafik yeni toparlanıyor, son 1 günlük data tek
başına yeterli sinyal değil).

Şüpheli (büyük + idx_scan=0):

| Index | Size | Table |
|---|---|---|
| recipes_search_gin | 792 KB | recipes |
| recipes_slug_trgm_idx | 520 KB | recipes |
| recipe_ingredients_name_trgm_idx | 472 KB | recipe_ingredients |
| recipes_title_trgm_idx | 424 KB | recipes |

Bunlar fulltext + fuzzy search için; `/tarifler` sayfasında
`websearch_to_tsquery` + pg_trgm similarity kullanıyor (kod mevcut). Prod'da
test edilip gerçekten sıfır trafik alıyor mu ya yeni reset olmuş mu
doğrulanmalı. Şu an erken, drop yok.

### 5.4 FK index eksiklik kontrolü

Audit script'te FK-üzerinde-index-eksiği kontrolü YOK (Prisma composite
index'leri zaten tanımlamış görünüyor). Faz 6 olarak pg_stat_user_indexes +
pg_index birleşiminde manuel kontrol eklenebilir; şimdilik Prisma schema
gözden geçirildi (`recipe_ingredients`, `recipe_steps`, `variations`,
`notifications`, `collections`, `meal_plan_items`, `recipe_photos` hepsinde
FK+sort composite index var).
